# Harbor Parity Experiment

This fork adds `infer_codex.py` to run CL-bench inference using the **Codex CLI agent inside Docker**, enabling a fair apples-to-apples parity comparison with the [Harbor clbench adapter](https://github.com/harbor-framework/harbor/tree/main/adapters/clbench).

## Why a separate script?

The original `infer.py` calls the OpenAI API directly. The Harbor adapter runs the Codex CLI agent inside a Docker container, which introduces file I/O overhead (reading messages from `/app/messages/`, writing answer to `/app/result.json`). To measure the true Harbor overhead, both sides must use the same agent, model, and execution environment.

## Alignment with Harbor adapter

| Dimension | Harbor adapter | This fork (`infer_codex.py`) |
|-----------|---------------|------------------------------|
| Agent | `codex@0.118.0` | `@openai/codex@0.118.0` (same) |
| Codex command | `codex exec --dangerously-bypass-approvals-and-sandbox --skip-git-repo-check --model {model} --json --enable unified_exec` | identical |
| Execution env | Docker container | Docker container (same image) |
| Messages | `/app/messages/NN_role.md` | `/app/messages/NN_role.md` (same) |
| Output | `/app/result.json` -> `{"output": "..."}` | reads same file, with `response.txt` fallback for older runs |
| Judge | `eval.py` with `gpt-4o-mini` | same rubrics and binary formula; parser and prompt tolerance hardened |

## Prerequisites

- Docker installed and running
- Python 3.8+
- `pip install openai tqdm`
- `OPENAI_API_KEY` set
- `OPENAI_BASE_URL` (optional, for custom endpoints)

## Reproduce parity experiment (gpt-5.2, 3 runs)

```bash
export OPENAI_API_KEY=<your_key>
export OPENAI_BASE_URL=<your_base_url>   # optional
export JUDGE_MODEL=gpt-4o-mini

# Run 3 independent trials with codex + gpt-5.2
for run in 1 2 3; do
  python infer_codex.py \
    --model gpt-5.2 \
    --input CL-bench_parity50.jsonl \
    --output outputs/codex_gpt-5.2_parity50_run${run}.jsonl \
    --workers 2

  python eval.py \
    --input outputs/codex_gpt-5.2_parity50_run${run}.jsonl \
    --output outputs/codex_gpt-5.2_parity50_run${run}_graded.jsonl \
    --judge-model gpt-4o-mini \
    --workers 5
done
```

The first run builds the Docker image automatically. Subsequent runs use `--skip-build` to reuse it:

```bash
python infer_codex.py --model gpt-5.2 --input CL-bench_parity50.jsonl \
    --output outputs/codex_gpt-5.2_parity50_run2.jsonl --workers 2 --skip-build
```

## Results

Scores are reported as `mean ± sample SEM`.

| Side | Runs | Solving Rate |
|------|------|--------------|
| Original (`infer_codex.py`) | 10%, 14%, 12% | 12.00% ± 1.15% |
| Harbor adapter | 12%, 8%, 12% | 10.67% ± 1.33% |

Run ranges overlap (original 10-14%, Harbor 8-12%), so the adapter is classified as matching by Harbor parity criteria.

## Evaluation compatibility

The parity fork keeps the CL-bench task rubrics and the binary scoring formula unchanged: a task succeeds only when all rubric requirements are satisfied. The `eval.py` changes are compatibility hardening:

- Robustly parse judge outputs when the judge returns fenced JSON, extra prose around JSON, or an `Overall Score` fallback.
- Instruct the judge to tolerate equivalent formatting, labels, whitespace, JSON key order, and Markdown presentation unless the rubric explicitly requires exact visible syntax.
- Keep failed parse attempts as score 0 after retries, preserving all-or-nothing grading.

These changes match the Harbor adapter evaluator behavior and are intended to avoid parser or presentation artifacts, not to add partial credit.

The scoring-adjacent tolerance rules are intentionally limited to cases where the rubric requirement is substantively satisfied but the judge output/parser or presentation style differs. They do not remove any rubric item, change the final binary threshold, add partial credit, or mark missing content as passing. If a rubric explicitly requires exact syntax, exact wording, exact formatting, a named field, or a specific implementation detail, the judge is still instructed to enforce that requirement.

## Parity set

`CL-bench_parity50.jsonl` contains 50 tasks sampled from the parity300 subset (indices listed in `parity300_indices.txt`). These correspond to Harbor task IDs `clbench-task-{N}` as generated by the Harbor adapter.

## Modifications to original benchmark

| File | Change | Reason |
|------|--------|--------|
| `infer_codex.py` | Codex-in-Docker inference runner that writes `/app/result.json` and reads the same result file, with compatibility fallback for older `/app/response.txt` runs | Match Harbor adapter execution and file I/O |
| `eval.py` | Harden judge-output parsing and add rubric-equivalence tolerance to the judge prompt; task rubrics and binary score formula are unchanged | Avoid parse failures or over-strict formatting failures that do not reflect the task rubric |
| `infer.py` | Preserve direct API inference while keeping output/resume behavior compatible with parity runs | Keep original API baseline runnable |
| `.gitignore` | Ignore local parity JSONL inputs, helper scripts, and generated outputs | Keep the parity fork working tree clean during local reproduction |
| `HARBOR.md` | New file | This document |

The scoring formula remains `num_success / total_samples`. No task rubrics were edited.

The fork history also contains upstream documentation/site assets such as `CNAME`, `docs/index.html`, `assets/four-concepts.png`, and `clbench-paper.pdf`. These files are not used by `infer_codex.py`, `infer.py`, `eval.py`, or the Harbor parity commands above, and they are not part of the scoring path. They should be treated as unrelated repository content rather than parity harness changes.
